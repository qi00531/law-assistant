# Homepage Contextual Follow-up And Notes Design

## Context

Law Assistant 当前首页问答已经具备基础的结构化回答能力，但核心学习链路仍然不稳定，和目标体验不一致。

当前主要问题：

1. 首页视觉信息过多，输入框上方仍残留“工作台”和解释性文案，不够极简。
2. 输入区仍包含辅助说明、字符计数等次要信息，不符合“只留输入框和发送按钮”的要求。
3. 多轮提问虽然有时间线雏形，但继续追问没有把前文上下文一起发给后端，结果仍然接近单轮问答。
4. 每轮结果后的引导词已经不需要，但当前页面没有把“直接继续输入即可带上下文”作为主交互。
5. 法条展示只有条号或法条名称，没有明确展示法条正文。
6. “已经理解”动作尚未落到每轮结果卡片底部，也没有与自动保存动作打通。
7. 首页保存调用的是后端接口，但卡片库页仍在读取前端 mock 数据，导致用户感知为“依旧没办法保存卡片”。

本轮改造聚焦首页问答主链路与卡片库真实数据接入，不扩展到新的页面或复杂会话系统。

## Goals

1. 首页问答页去掉输入框上方“工作台”和多余说明，整体维持极简风格。
2. 左侧导航底部提供真实可用的深浅主题切换按钮，首页头部不再重复放切换入口。
3. 输入区只保留输入框和发送按钮，保留输入框滚动能力，不再展示其他辅助信息。
4. 首页首屏不再保留任何大标题欢迎区、品牌说明区或示例引导区，直接进入结果区与输入区。
5. 多轮提问结果在同一页连续累积，用户可以向下滚动翻阅全部历史结果。
6. 用户继续输入时，系统自动把前文结构化结果打包为上下文，一并交给后端生成下一轮答案。
7. 每轮结果卡片底部提供“已经理解”按钮，点击后自动保存该轮卡片。
8. 结果中的相关法条需要展示具体条文内容，而不只是法条标题或条号。
9. 卡片库页改为读取真实后端数据，让首页保存后的卡片能立即在卡片库中看到。

## Non-Goals

1. 不新增独立聊天页或新的首页子路由。
2. 不引入流式输出、逐字生成或打字机效果。
3. 不实现长期会话存储或多用户会话管理。
4. 不重构 ReviewMode 内部学习阶段逻辑。
5. 不对学习记录页做额外视觉重构。

## User Experience

### Homepage Layout

首页问答页改成更克制的结构：

1. 保留左侧导航。
2. 主内容区只保留可滚动结果区和底部固定输入区。
3. 删除截图红框内的全部内容，包括顶部品牌头部卡、页面大标题、说明文案和首屏引导块。
4. 首页不再保留任何首屏占位视觉区，未提问时也直接展示输入区。
5. 不删除滚动条，用户仍通过自然滚动翻阅多轮结果。

### Theme Toggle

深浅主题切换入口移动到左侧导航底部，并作为真实按钮生效。

要求：

1. 按钮始终可见。
2. 折叠侧栏时仍保留图标级可点击入口。
3. 首页头部不再出现重复的主题切换按钮。

### Composer

底部输入区只保留两个元素：

1. 输入框
2. 发送按钮

移除内容包括：

1. 工作台标签
2. Enter/Shift + Enter 提示
3. 字数计数
4. AI 学习提示文案
5. 输入框下方免责声明
6. 快速提问或首屏说明区域

输入框继续支持：

1. 自动增高
2. Enter 提交
3. Shift + Enter 换行
4. 内容超长时内部滚动

### Multi-turn Results

首页采用“单输入框，多轮结果累积在同一页”的模式。

交互规则：

1. 页面初始时不展示头部介绍，只保留空结果区和底部输入区。
2. 第一次提问后，在结果区追加第一张结果卡片。
3. 后续每次继续输入，都在结果区底部追加新的结果卡片。
4. 历史结果不覆盖、不跳页、不折叠到其他页面。
5. 用户通过滚动翻阅之前各轮结果。

### Follow-up Behavior

每轮结果卡片后不再显示引导词。

继续追问的方式改为：

1. 用户直接在同一个输入框里继续输入。
2. 系统自动把前面各轮问题和结构化结果打包成上下文。
3. 新一轮答案基于完整上下文生成，但依然只渲染为一张新的结构化结果卡片。

页面要传达的心智是：

“不需要点引导词，继续输入即可在当前学习链路上追问。”

### Result Card Actions

每轮结果卡片底部提供一个主动作按钮：

1. `已经理解`

点击后行为：

1. 自动把当前这一轮结果保存为一张卡片。
2. 保存成功后按钮进入已保存状态，避免重复提交。
3. 若保存失败，在当前卡片附近给出明确错误反馈。

“已经理解”是单轮卡片级动作，不影响整页继续提问。

### Statute Display

结果卡片中的法条模块要展示：

1. 法律名称
2. 条号
3. 条文正文

不再只显示：

1. `《中华人民共和国民法典》第五百二十七条`

而是要同时显示该条的具体内容，方便用户直接阅读和比对。

## Information Architecture

首页问答区调整为两层：

### 1. Scrollable Results

负责展示：

1. 多轮结果卡片
2. 每轮结果内的结构化模块
3. 每轮卡片底部的“已经理解”按钮
4. 当前卡片的保存反馈

### 2. Fixed Composer

负责：

1. 输入问题
2. 发起提问
3. 展示提交中状态

与之前不同的是，页面不再保留单独的顶部控制条或会话动作区作为主入口。

## Data Model

## Ask Request

`/api/ask` 从单轮请求扩展为支持上下文：

1. `question`
2. `history`

其中 `history` 按时间顺序携带已有轮次，每项至少包含：

1. `question`
2. `answer`
   - 即结构化学习结果

后端统一负责将这些历史轮次组织为 prompt，而不是由前端自行拼自然语言大段文本。

## Ask Response

结构化响应保持现有主体不变，但法条字段需要升级为可展示正文的结构。

推荐从：

1. `statutes: string[]`

改为：

1. `statutes: StatuteItem[]`

`StatuteItem` 至少包含：

1. `title`
2. `article`
3. `content`

如果上游模型暂时不能稳定产出完整结构，后端也可以先接受兼容输入，再转换成统一响应结构。

## Timeline Item

前端每轮结果项至少维护：

1. `id`
2. `question`
3. `relation`
4. `createdAt`
5. `result`
6. `modules`
7. `saveState`
   - `idle`
   - `saving`
   - `saved`
   - `error`
8. `saveFeedback`

其中 `saveState` 用于驱动“已经理解”按钮状态。

## Notes

卡片保存分为两种来源：

1. 首页单轮“已经理解”保存
2. 现有整轮会话汇总保存

本轮至少保证第一种来源可用，并且卡片库能读取到真实数据。

## Behavior Changes

### Homepage Submit Logic

首页提交逻辑调整为：

1. 首轮提问时，`history` 为空。
2. 后续提问时，将当前时间线中每一轮的 `question + result` 一起传给后端。
3. 后端基于上下文返回新的结构化结果。
4. 前端将新结果追加到时间线末尾。

### Save Per Result

点击某张结果卡片底部的“已经理解”时：

1. 以前轮问题为标题基础生成单卡片 payload。
2. 调用 `POST /api/notes`。
3. 成功后更新当前卡片的按钮状态与反馈文案。

### Notes Page Source Of Truth

卡片库页改为以真实后端数据为准。

最小改造要求：

1. 列表读取 `GET /api/notes`
2. 详情读取 `GET /api/notes/{id}`
3. 不再把 mock 数据作为唯一来源

如果部分视觉字段后端暂未提供，可以在前端按真实数据做轻量推导，但数据主源必须切到后端。

## Error Handling

### Ask Failures

若带上下文的问答失败：

1. 不清空现有时间线
2. 在输入区上方或结果区下方显示错误提示
3. 保留用户输入，便于再次发送

### Save Failures

若“已经理解”保存失败：

1. 当前卡片按钮恢复可点击
2. 显示明确错误信息
3. 不影响用户继续追问

### Notes Fetch Failures

若卡片库获取真实数据失败：

1. 展示明确的空状态或错误状态
2. 不伪装成“没有数据”

## Acceptance Criteria

完成后应满足：

1. 首页输入框上方不再显示“工作台”及多余说明。
2. 截图红框中的顶部头部卡、标题区、说明区和首屏引导区全部删除。
3. 左侧导航底部主题按钮可真实切换深浅主题。
4. 输入区只剩输入框和发送按钮。
5. 多轮提问结果会持续累积在同一页，可通过滚动翻阅。
6. 继续输入时会自动携带前文上下文回答。
7. 每轮结果卡片底部有“已经理解”按钮。
8. 点击“已经理解”后会自动保存当前轮卡片。
9. 法条模块会显示具体条文正文。
10. 卡片库能看到首页新保存的卡片。
11. 保存失败时用户能看到明确反馈，而不是“像没保存成功”。

## Recommendation

按“极简首页 + 后端统一处理多轮上下文 + 单轮自动保存 + 卡片库接入真实数据”的方式改造。这是满足当前需求且后续最容易继续扩展的路径。
