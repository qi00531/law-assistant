# Homepage Structured Learning Timeline Design

## Context

Law Assistant 当前首页问答已经接入真实 `/api/ask`，但交互仍然更接近“单结果页 + 单次回答”。这与产品目标不完全匹配。

当前主要问题：

1. 首页结果区仍是单次替换，不是连续学习过程。
2. 引导词点击会覆盖当前输入或替换当前结果，不符合“继续追问”的学习节奏。
3. 复习入口放在页面顶部，触发条件依赖“当前会话重复提问”，不符合“已保存知识点再复习”的心智。
4. 保存笔记仍偏向保存单条结果，不是对整轮学习过程进行沉淀。
5. 结果区和输入区的布局不够稳定，无法形成“中间滑动、底部固定输入”的长期使用模式。

本轮重构只聚焦首页问答主链路，不同时处理学习记录页和卡片库页的交互改造。

## Goals

将首页问答重构为“结构化学习时间线”，让体验更接近持续学习过程，而不是单次回答页。

完成后首页应具备这些能力：

1. 初始态简洁，不展示示例结果，只展示欢迎区、引导词和输入框。
2. 提问后进入连续学习时间线，每次提问生成一条新的结构化学习记录。
3. 点击引导词时，不覆盖原有内容，而是作为“基于上一题继续追问”的新记录加入时间线。
4. 页面中间结果区可滚动，底部输入框固定。
5. 若当前提问与已保存笔记高度相似，则在对应结果块内提供复习入口。
6. 点击保存时，将整轮时间线汇总为一张笔记，而不是只保存最后一条回答。
7. 保持现有冷白背景、轻面板、结构化卡片的视觉语言，不改成聊天气泡界面。

## Non-Goals

本轮明确不做：

1. 不重构学习记录页顶部四卡和筛选行为。
2. 不重构卡片库整体布局。
3. 不新增路由，不拆成新的问答子页面。
4. 不引入多用户会话存储。
5. 不引入流式响应或实时生成动画。
6. 不改动 ReviewMode 组件内部的学习阶段逻辑，只调整首页入口位置和触发条件。

## User Experience

### Initial State

首页首次进入时，只显示：

1. 品牌欢迎区
2. 快速引导词
3. 固定输入框

不显示任何默认案例卡片，不展示预置学习结果。

### Learning Timeline Mode

用户发送第一个问题后，首页切换为“学习时间线模式”。

页面结构：

1. 顶部保留轻量页面头部
2. 中间为可滚动的学习时间线容器
3. 底部为固定输入框

每次提问会生成一条新的学习记录块，而不是替换旧内容。

每条学习记录块包含：

1. 问题标题
2. 结构化结果模块
3. 可选的复习入口
4. 当前条目的辅助操作

### Suggested Follow-up Questions

引导词不再覆盖当前输入内容，也不替换当前结果。

点击引导词后：

1. 生成一条新的问题记录
2. 该问题与上一条记录建立“关联追问”关系
3. 在时间线中显示为新的学习块

视觉上可通过较轻的“继续追问”标识体现它是从上一题延伸出来的，但不做聊天箭头或气泡关系线。

### Review Entry

首页不再根据“当前会话重复提问”触发复习提示。

新的复习提示规则：

1. 当前问题与卡片库中已保存笔记做相似度比较
2. 若相似度达到阈值，则在当前学习记录块内部显示轻提示
3. 提示内容强调“这个知识点你已经保存过，可以直接进入复习”
4. 点击后在当前记录块附近展开 ReviewMode

ReviewMode 不再放在页面顶部。

### Save Note

保存笔记改为“整轮汇总保存”。

一轮学习时间线由：

1. 首问题
2. 后续关联追问
3. 各自的结构化结果

共同组成。

点击“保存笔记”时：

1. 以前一轮首问题为标题基础
2. 将后续关联追问的结果压缩为补充摘要
3. 合并概念、要件、法条、误区和易混点，生成一张汇总笔记

这样保存结果更像“学习后的沉淀”，而不是单条问答快照。

## Information Architecture

首页问答区拆成三个层：

### 1. Fixed Composer

固定在页面底部，负责：

1. 输入问题
2. 提交问题
3. 显示加载状态
4. 保持输入体验连续

### 2. Scrollable Timeline

用于展示整轮学习过程，包括：

1. 主问题结果
2. 关联追问结果
3. 当前记录内的复习入口
4. 当前记录级别的轻动作

### 3. Session Actions

面向整轮学习会话，而不是单条卡片：

1. 保存本轮笔记
2. 清空本轮
3. 继续追问

## Data Model

首页前端新增会话级状态，而不再只保存 `activeQuestion` 和一组 `learningModules`。

### Timeline Item

每一条学习记录至少包含：

1. `id`
2. `question`
3. `relation`
   - `root`
   - `follow_up`
4. `result`
   - 后端 `/api/ask` 返回的结构化结果
5. `modules`
   - 前端映射后的展示模块
6. `relatedNote`
   - 若命中已保存笔记，则记录对应笔记信息
7. `createdAt`

### Session Summary

会话级别维护：

1. 当前时间线数组
2. 当前输入草稿
3. 提交中状态
4. 错误状态
5. 当前展开的复习项
6. 当前会话是否已保存

## Behavior Changes

### Replace Single Result Mode

当前 `isResultMode` 只基于是否有 `activeQuestion`。

重构后：

1. `isResultMode` 基于时间线是否为空
2. 页面不再只依赖单个 `activeQuestion`
3. 所有结果渲染来自时间线数组

### Similarity Logic

问题相似度比较分两种用途：

1. 关联当前时间线中的问题，用于标识 follow-up 关系
2. 对比已保存笔记，用于给出复习入口

首页不再因为“当前会话中重复问过”而提示复习。

### Example Chips

引导词点击后不覆盖当前草稿内容。

行为改为：

1. 若页面还未开始会话，则直接发起首个问题
2. 若页面已有时间线，则生成一条 follow-up 问题

## Notes Integration

保存笔记调用仍使用 `POST /api/notes`，但 payload 来源改为“整轮会话汇总”。

前端会增加一个汇总函数：

1. 选取首问题作为基础标题
2. 聚合多条结果中的概念、要件、法条、误区和 confusions
3. 生成一个简洁 summary
4. 自动推断标签

本轮不要求卡片库立即实时刷新为后端数据源，但首页保存成功反馈必须明确。

## Theming

首页加入真实可点击的深浅模式切换按钮。

要求：

1. 支持 `light` 和 `dark`
2. 切换按钮替代当前占位背景切换按钮
3. 主题状态保存在前端本地
4. 首页问答、时间线、输入框、结构化卡片在两种主题下都可读

本轮先把首页问答体验和主题切换打通，不要求全站每一处页面都完成精修。

## Visual Design

视觉继续保持：

1. 冷白背景为主
2. 深色面板用于重点内容
3. 绿色作为轻高亮
4. 强留白
5. 非聊天气泡式布局

时间线里的每条学习记录块要：

1. 明确区分问题头和结构化内容
2. 允许纵向滚动阅读
3. 保持输入框固定时的稳定层级

## Error Handling

### Ask Failure

若 `/api/ask` 失败：

1. 不向时间线插入半成品卡片
2. 在输入框附近显示轻量错误提示
3. 保留当前输入内容

### Save Failure

若 `/api/notes` 失败：

1. 不清空当前时间线
2. 显示“保存失败，请稍后重试”
3. 允许用户再次点击保存

### Review Lookup Failure

若本地笔记相似度判断失败或没有匹配结果：

1. 不显示复习入口
2. 不影响当前学习结果展示

## Implementation Scope

本轮主要涉及：

1. `frontend/src/pages/HomePage.vue`
2. `frontend/src/components/home/*`
3. `frontend/src/lib/api.ts`
4. `frontend/src/utils/*`
5. 可能增加首页会话和笔记汇总工具文件
6. 主题切换相关的轻量 composable 或 store

不修改路由，不新增新页面。

## Acceptance Criteria

完成后应满足：

1. 首页初始态没有默认示例结果
2. 输入第一个问题后进入结构化学习时间线
3. 中间结果区可滚动，输入框固定
4. 引导词点击不会覆盖旧内容，而是生成关联追问
5. 复习入口只在命中已保存笔记时出现
6. ReviewMode 出现在对应记录块附近，而不是页面顶部
7. 保存笔记会汇总整轮内容
8. 深浅模式按钮可真实切换主题
9. 问答失败和保存失败都有清晰反馈

## Risks

1. 若首页一次展示过多结构化卡片，时间线可能过长，需要控制每条记录的默认展开密度。
2. 若整轮汇总逻辑过于简单，保存笔记质量可能不稳定，需要后续继续优化摘要合并策略。
3. 深浅模式若只完成首页而未覆盖所有共享组件，可能出现局部视觉不一致，需要在实现阶段设定边界。

## Recommendation

按“结构化学习时间线”方案执行，先完成首页链路，再在下一轮处理学习记录与卡片库的同步重构。
